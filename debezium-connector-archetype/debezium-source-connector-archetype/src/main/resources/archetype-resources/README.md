
The coordinator runs the snapshot source first, then hands off to the streaming source. Both produce `ChangeRecordEmitter` instances, which the `EventDispatcher` converts to `SourceRecord`s and pushes onto the queue. Kafka Connect calls `poll()` from a separate thread; `doPoll()` drains whatever is buffered.

## Key concepts

**Partition.** The key under which the framework stores your offsets in Kafka Connect's offset topic. The default implementation maps a single server name (the `topic.prefix` config value) to one offset slot. If your source has multiple independently resumable streams (shards, replicas, ...), model each one as a separate `Partition` via the inner `Provider`.

**OffsetContext.** Your connector's in-memory representation of "where am I in the source." Right now it tracks `position` as a `long`; replace that with whatever your source needs (LSN, log filename plus byte position, change vector, timestamp, ...). Whatever `getOffset()` returns is what gets persisted, and what `${connectorName}OffsetLoader` must be able to deserialize on restart.

**SourceInfo and SourceInfoStructMaker.** Every change event has a `source` block describing where the event came from. `${connectorName}SourceInfo` is the data holder; `${connectorName}SourceInfoStructMaker` builds the Kafka Connect `Schema` for it. When you add a field to one, register it in the other.

**DataCollectionId.** Identifies a single table or collection by string name. The dispatcher uses it for topic routing via the configured `TopicNamingStrategy`, and for filtering.

**DatabaseSchema.** A registry of `DataCollectionSchema` entries keyed by `DataCollectionId`. Populate it during snapshot or initial discovery so the dispatcher knows the shape of each collection.

**ChangeRecordEmitter.** What the snapshot and streaming sources actually emit. It carries the partition, the offset context, a key `Struct`, and a value `Struct`. The dispatcher receives it and produces the final `SourceRecord`.

**SnapshotMode.** Defined as an enum on `${connectorName}ConnectorConfig` with `INITIAL` and `NEVER`. `INITIAL` means "snapshot only if no offset exists." Add more modes here if your source supports them (e.g., `SCHEMA_ONLY`, `WHEN_NEEDED`).

## What to implement, in order

The skeleton compiles, but emits no events until you fill in the steps below. Doing them in this order keeps each step short and testable:

1. **`${connectorName}ConnectorConfig`** — declare every configuration `Field` your connector accepts (connection URL, credentials, batching limits, ...). Add each one to `ALL_FIELDS` and to a `Field.group(...)` call in `configDef()`. Add validation where it helps.

2. **`${connectorName}OffsetContext`** — replace the placeholder `position` field with a real model of your source's offset. Update `getOffset()` to serialize it, and update `${connectorName}OffsetLoader` to read it back. This is the contract that makes restarts work.

3. **`${connectorName}SourceInfo` and `${connectorName}SourceInfoStructMaker`** — add whatever per-event source metadata you want to expose (LSN, transaction id, server name, ...). Keep both files in sync — fields exposed on `SourceInfo` must appear in the struct maker.

4. **`${connectorName}DatabaseSchema`** — load and register `DataCollectionSchema` objects for the tables or collections you want to capture. This usually happens during snapshot setup or initial connection.

5. **`${connectorName}SnapshotChangeEventSource.doExecute(...)`** — read existing data from the source and emit a `${connectorName}ChangeRecordEmitter` per row with operation `READ`. Use the `EventDispatcher` that was injected through the factory. Update the offset context as you go so a mid-snapshot restart can resume.

6. **`${connectorName}StreamingChangeEventSource.execute(...)`** — connect to your source's change feed (binlog, WAL, change stream, polling, ...) and emit a `${connectorName}ChangeRecordEmitter` per change. Handle `CREATE`, `UPDATE`, and `DELETE` (and `TRUNCATE` if your source has it). Update the offset context with every event.

## What you usually don't need to touch

- **`${connectorName}SourceConnector`** — the one-task-per-connector pattern is correct for almost every source.
- **`${connectorName}ConnectorTask.start(...)`** — the wiring is standard. You generally only swap in different *inputs* (e.g., a richer `Partition.Provider`); the queue, dispatcher, and coordinator construction is the same across connectors.
- **`${connectorName}Partition`** — fine as-is unless your source has multiple resumable streams.
- **`${connectorName}ErrorHandler`** — the passthrough behavior is correct unless your source has specific error categories that should be retried instead of fatal.
- **`${connectorName}EventMetadataProvider`** — the defaults return what JMX metrics need.
- **`Module`** — the version constant is wired from your project's POM at generation time. You only edit `name()` if you want a different SLF4J/JMX context name.

## Build and run

```bash
mvn package
```

Produces an uber-jar at `target/${artifactId}-${version}.jar` containing all the runtime dependencies Kafka Connect needs. To run it:

1. Copy the jar into a directory listed in Kafka Connect's `plugin.path`.
2. Restart Connect (or use the REST API to install if your distribution supports it).
3. POST a connector config to Connect's REST API. Minimal example:

```json
{
  "name": "${connectorName.toLowerCase()}-source",
  "config": {
    "connector.class": "${package}.${connectorName}SourceConnector",
    "tasks.max": "1",
    "topic.prefix": "my-server",
    "snapshot.mode": "initial"
  }
}
```

Add any other `Field`s you declared in `${connectorName}ConnectorConfig` to the `config` block.

## Optional next steps

- **`Snapshotter` SPI** — the modern, finer-grained way to control snapshots. Implement `io.debezium.spi.snapshot.Snapshotter` and let the framework call you back at each phase instead of overriding the snapshot source directly.
- **Signal channel** — support `io.debezium.pipeline.signal.SignalAction` so users can drive ad-hoc snapshots and pauses from outside.
- **Notification service** — already wired in `${connectorName}ConnectorTask`. Emit framework notifications for milestones (snapshot started, snapshot completed, ...) so external systems can react.
- **JMX metrics** — base classes register standard metrics automatically. Add custom metrics by extending the metrics factory passed to the coordinator.
- **Integration tests** — `debezium-core` provides `AbstractConnectorTest` and friends. A real test usually spins up a Docker container running your source database and exercises the full pipeline.

== About Data Lineage and OpenLineage

Data lineage tracks the flow of data through various systems, transformations, and processes.
This information provides you with visibility into where data originates, how it moves, and what dependencies exist in the data pipeline.
Insights into data lineage is crucial for the following activities:

* Data governance and compliance
* Impact analysis when making changes
* Debugging data quality issues
* Understanding data dependencies

https://openlineage.io/[OpenLineage] is an open standard for data lineage that provides a unified way to collect and track lineage metadata across multiple data systems.
The specification defines a common model for describing datasets, jobs, and runs, simplifying the process of building comprehensive lineage graphs across heterogeneous data infrastructures.

For more information, see the OpenLineage https://openlineage.io/[website] and https://openlineage.io/docs/[documentation].

[id="debezium-openlineage-run-events"]
== Run events

{prodname} emits OpenLineage run events to track connector status throughout its lifecycle, from initialization through shutdown.
These events provide visibility into connector health and enable real-time monitoring of data pipeline operations.

The connector emits run events after the following status changes:

START:: Reports connector initialization.
RUNNING:: Emitted periodically during normal streaming operations and during processing individual tables.
These periodic events ensure continuous lineage tracking for long-running streaming CDC operations.
COMPLETE:: Reports that the connector shut down gracefully.
FAIL:: Reports that the connector encountered an error.

[id="debezium-openlineage-job-mapping"]
=== OpenLineage job mapping

The {prodname} connector is mapped to an OpenLineage *Job*, which includes the following elements:

Name:: A job inherits its name from the {prodname} `topic.prefix`, combined with the task ID (for example, `inventory.0`).
Namespace:: Inherited from `openlineage.integration.job.namespace`, if specified; otherwise defaults to the value of the `topic.prefix`.
{prodname} connector version:: The version of the {prodname} connector that is running and generating lineage events.
Complete connector configuration:: All connector configuration properties, enabling full reproducibility and debugging of the data pipeline.
Job metadata:: Description, tags, and owners.

// Type: concept
[id="debezium-openlineage-dataset-mapping-for-source-connectors"]
=== Dataset mapping for source connectors

The following dataset mappings are possible:

Input Datasets::
Represent the database tables that {prodname} is configured to capture changes from.
The OpenLineage integration automatically creates input datasets based on the connector configuration.
The integration applies the following principles when it creates dataset mappings:

* Each table that the connector monitors becomes an input dataset.
* Each dataset captures schema information for the corresponding source table, including the name and data type of each column.
* DDL changes in the source table are reflected dynamically in the dataset schema.

Output datasets::
Represent the Kafka topics where CDC events are written.
For Kafka Connect deployments, output datasets are emitted when you apply the OpenLineage single message transformation (SMT).
For {prodname} Server deployments, output datasets are automatically captured.
+
Output dataset mappings are created according to the following principles:

* Each Kafka topic that the connector produces becomes an output dataset.
* The output dataset captures the complete CDC event structure, including metadata fields.
* The name of the dataset is based on the connector's topic prefix configuration.

// Type: concept
[id="debezium-openlineage-dataset-mapping-for-sink-connectors"]
=== Dataset mapping for sink connectors

For sink connectors, the data flow is reversed compared to source connectors.

Input datasets::
Represent the Kafka topics that the sink connector reads from.
These topics typically contain CDC events from {prodname} source connectors.
The following principles apply when defining input datasets:
+
* Each Kafka topic that the sink connector consumes represents an input dataset.
* The input dataset specifies the Kafka topic schema and metadata.
* The namespace format follows `kafka://bootstrap-server:port`, where the bootstrap server is specified via the `openlineage.integration.dataset.kafka.bootstrap.servers` property.

Output datasets::
Represent the target databases or collections where the sink connector writes data.
+
The following principles apply when defining output dataset mappings:
+
* Each target database table or collection represents an output dataset.
* The output dataset specifies the schema information for the target database.
* The namespace format depends on the target database system.
For more information, see xref:dataset-namespace-formatting[Dataset namespace formatting].

[id="debezium-openlineage-streams-deployment-procedure"]
=== Deploying the OpenLineage integration on {StreamsName}

You can deploy the {prodname} OpenLineage integration in a {StreamsName} environment to enable lineage tracking for your {prodname} connectors.

In {StreamsName} environments, Kafka Connect runs as an immutable, containerized workload managed by the Streams operator.
You cannot modify running Kafka Connect pods by copying files or adding libraries after deployment.
To enable the OpenLineage integration, you must complete the following tasks:

. Specify the OpenLineage Java client as a Maven dependency in the Kafka Connect build configuration.
. Provide OpenLineage runtime configuration in a ConfigMap.
. Mount the configuration into the Kafka Connect pod.

When you deploy OpenLineage on {StreamsName}, consider the following deployment characteristics:

Build-time dependency inclusion::
You include the OpenLineage Java client as a Maven dependency in the KafkaConnect custom resource.
During deployment, the Streams operator automatically builds a Kafka Connect image that contains the required OpenLineage artifacts.

Maven repository access::
The Kafka Connect build process must be able to access a Maven repository that hosts the OpenLineage artifacts.
This repository can be a remote repository, such as Maven Central, or an internal mirror in restricted or disconnected environments.

Externalized runtime configuration::
OpenLineage runtime configuration cannot be defined inline in the KafkaConnect custom resource alone.
You must provide the `openlineage.yml` configuration file by using an external Kubernetes resource, such as a `ConfigMap`, and mount it into the Kafka Connect container.

Explicit configuration discovery::
The OpenLineage Java client does not automatically discover configuration files in containerized Kafka Connect deployments.
To ensure that your deployment can read the configuration, define the `OPENLINEAGE_CONFIG` environment variable in the Kafka Connect custom resource to specify the path to the mounted configuration file.

Cluster-level scope::
OpenLineage configuration applies at the Kafka Connect cluster level.
All connectors in the cluster share the same configuration, and changes to the OpenLineage configuration require restarting the Kafka Connect pods to take effect.
This approach centralizes OpenLineage behavior while enabling you to define a unique job identity for each connector.

[id="debezium-openlineage-streams-add-dependencies"]
==== Adding OpenLineage dependencies to your {StreamsName} Kafka Connect image

Now that you understand the deployment model, the first step toward enabling lineage tracking is to specify the OpenLineage client as a dependency in your Kafka Connect custom resource.

The OpenLineage Java client is a Java library that is available on Maven Central.

To include OpenLineage in your Kafka Connect deployment, you specify the OpenLineage Java client as a Maven dependency in your KafkaConnect custom resource.
The Streams operator automatically retrieves and installs the artifacts, enabling you to track lineage without manual file management.
When you list OpenLineage artifacts in the `spec.build.plugins` section of the `KafkaConnect` custom resource, the operator automatically resolves and downloads the required artifacts from a Maven repository and installs them into the Kafka Connect plugin path.

It is not necessary to manually download or extract OpenLineage archives.

.Procedure

. Add the OpenLineage artifact by inserting the following lines in the `spec.build` section of the YAML that defines your Kafka Connect CR:
+
[source,yaml,subs="+attributes"]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaConnect
metadata:
  name: debezium-connect
spec:
  version: 3.7.0
  bootstrapServers: my-cluster-kafka-bootstrap:9092
  build:
    output:
      type: docker
      image: my-registry/debezium-connect:latest
    plugins:
      - name: openlineage
        artifacts:
          - type: maven
            group: io.openlineage
            artifact: openlineage-java
            version: 1.15.0
----
+
The following list describes the purpose of select lines in the preceding Kafka Connect configuration example:
+
bootstrapServers::
Specifies the Kafka bootstrap address that Kafka Connect uses to connect to the Kafka cluster.
+
build::
Enables operator-managed container image builds for Kafka Connect.
+
build.output.image::
Specifies the name of the container image that the Streams operator generates and uses to deploy Kafka Connect pods.
+
build.plugins::
Declares additional plugins and dependencies to include in the Kafka Connect image.
+
plugins.name::
Provides a logical name for the plugin.
Kafka Connect uses the name for organization and does not affect runtime behavior.
+
artifacts.type = maven::
Indicates that the operator retrieves the artifact from a Maven repository during the image build.
You specify the artifact using Maven coordinates (groupId, artifactId, and version).
By default, the operator uses the Maven Central repository.
To use a different repository, set the `MAVEN_REPOSITORY` environment variable in `spec.template.connectContainer.env` in the `KafkaConnect` custom resource.
+
artifacts.group::
Specifies the Maven group ID of the OpenLineage Java client artifact.
+
artifacts.artifact::
Specifies the Maven artifact ID of the OpenLineage Java client.
+
artifacts.version::
Specifies the version of the OpenLineage Java client that the operator includes in the Kafka Connect image.

[NOTE]
====
To retrieve the OpenLineage artifacts, the Kafka Connect cluster must be able to access the configured Maven repository.
In a restricted or disconnected environment, configure access to a mirrored or internal repository.
====

[id="debezium-openlineage-streams-configure-behavior"]
==== Configuring OpenLineage behavior in {StreamsName}

To complete the deployment, you must make your OpenLineage configuration accessible to the Kafka Connect pods.

With OpenLineage included in your image, the next step is to configure how it behaves and where it sends lineage events.
The OpenLineage integration uses runtime configuration that is external to the KafkaConnect custom resource.
You cannot define this configuration inline in the Kafka Connect custom resource alone.
In {StreamsName} environments, you supply the OpenLineage configuration in an OpenShift `ConfigMap`, which is mounted into the Kafka Connect pod and referenced by the Kafka Connect custom resource.

.Procedure

. Create a ConfigMap that contains the OpenLineage configuration:
+
[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: openlineage-config
data:
  openlineage.yml: |
    transport:
      type: http
      url: http://openlineage-server:5000
----
+
The following list describes the purpose of select lines in the preceding OpenLineage ConfigMap example:
+
metadata.name::
Defines the name of the `ConfigMap` that provides the OpenLineage configuration.
You reference this name in the Kafka Connect custom resource.
+
data.openlineage.yml::
Defines the OpenLineage configuration file that you mount into the Kafka Connect container.
+
transport.type::
Specifies the transport mechanism that the OpenLineage client uses to emit lineage events.
+
transport.url::
Specifies the endpoint of the OpenLineage service that receives lineage events.

. Create the `ConfigMap` in the same namespace as the Kafka Connect cluster.

// Type: procedure
[id="debezium-openlineage-streams-mount-configuration"]
==== Mounting the OpenLineage configuration in {StreamsName} Kafka Connect

To make the OpenLineage configuration available to Kafka Connect, mount the `ConfigMap` into the Kafka Connect container and specify the configuration file location by using an environment variable.

[IMPORTANT]
====
As part of enabling the OpenLineage integration, you must explicitly specify the location from which Kafka Connect loads the OpenLineage configuration.
The `OPENLINEAGE_CONFIG` environment variable specifies where the configuration file is located, while the values defined in `openlineage.yml` specify how OpenLineage behaves.
These settings serve different purposes and you configure them independently.
====


The following configuration mounts the `openlineage.yml` file into the Kafka Connect container making it available to the OpenLineage client at startup.

.Procedure

. Add the following configuration to your KafkaConnect custom resource:
+
[source,yaml]
----
spec:
  template:
    connectContainer:
      env:
        - name: OPENLINEAGE_CONFIG
          value: /opt/openlineage/openlineage.yml
      volumeMounts:
        - name: openlineage-config
          mountPath: /opt/openlineage
    pod:
      volumes:
        - name: openlineage-config
          configMap:
            name: openlineage-config
----
+
The following list describes the purpose of select lines in the preceding example:
+
env.name = OPENLINEAGE_CONFIG::
Defines the environment variable used by the OpenLineage Java client to locate its configuration file.
The variable specifies where the OpenLineage client loads the specified configuration file from at startup.
It does not define or modify OpenLineage behavior.
+
env.value::
Specifies the absolute filesystem path to the OpenLineage configuration file, `openlineage.yml` within the Kafka Connect container.
The location of the `openlineage.yml` file is not fixed. 
In operator-managed {StreamsName} environments, you can mount the configuration file at any path, as long as you reference the path in the `OPENLINEAGE_CONFIG` environment, and Kafka Connect can read the file.
+
volumeMounts.mountPath::
Defines the directory inside the Kafka Connect container where you mount the `ConfigMap`.
The `mountPath` field is essential to placethe ConfigMap into the container, and making the configuration available.
+
volumes.configMap.name::
References the `ConfigMap` resource that contains the OpenLineage configuration file.

[id="debezium-openlineage-job-metadata"]
== Job metadata enrichment

To enhance the utility of the lineage data that it emits, you can add metadata that provides tags and owners information for each job.
Tags are custom, arbitrary metadata that you can attach to jobs to provide additional context to help classify the job.
Owners indicate who is responsible for the job.
Use the following formats to specify tags and owners metadata in the OpenLineage configuration properties:

Tags list format::
Specify tags as a comma-separated list of key-value pairs, as shown in the following example:
[source,properties]
----
openlineage.integration.job.tags=environment=production,team=data-platform,criticality=high
----

Owners list format::
Specify owners as a comma-separated list of name-role pairs, as shown in the following example:
[source,properties]
----
openlineage.integration.job.owners=John Doe=maintainer,Jane Smith=Data Engineer,Team Lead=owner
----

// Type: reference
[id="debezium-openlineage-kafka-connect-output-dataset-lineage"]
=== Kafka Connect output dataset lineage

To capture output dataset lineage in Kafka Connect, configure {prodname} to use the OpenLineage Single Message Transform (SMT):

[source,properties]
----
# Add OpenLineage transform
transforms=openlineage
transforms.openlineage.type=io.debezium.transforms.openlineage.OpenLineage

# Required: Configure schema history with Kafka bootstrap servers
schema.history.internal.kafka.bootstrap.servers=your-kafka:9092
----

The SMT captures detailed schema information about change events that {prodname} writes to Kafka topics.
The transformation captures schema data that includes the following items:

* Event structure (before, after, source, transaction metadata)
* Field types and nested structures
* Topic names and namespaces

// Type: procedure
[id="debezium-openlineage-verifying-the-integration"]
=== Verifying the integration

To verify that the OpenLineage integration is working correctly, complete the following steps:

.Procedure
1. Check the connector logs for OpenLineage-related messages.
2. If you configured HTTP transport, verify that events appear in your OpenLineage backend.
3. For testing purposes, you can configure console transport to view events directly in the logs, as shown in the following example:
+
[source,yaml]
----
transport:
  type: console
----

// Type: reference
[id="debezium-openlineage-common-issues"]
=== Common issues

Integration not working::
* Verify that `openlineage.integration.enabled` is set to `true`.
* Check that the path to the OpenLineage configuration file that is specified in the connector configuration is correct, and that {prodname} can access the target file.
* Ensure that the YAML in the OpenLineage configuration file is valid.
* Verify that all required JAR dependencies are present in the classpath.

Missing output datasets::

* Verify that you configured the connector to use the OpenLineage transformation.
* Check that you set the property `schema.history.internal.kafka.bootstrap.servers` in the connector configuration.

Connection issues::

* Verify that you specified the correct server URL and authentication information in the OpenLineage client configuration.
* Check the network connectivity between {prodname} and the OpenLineage server.

Dependency issues::

* Ensure that all required JAR files are present and their versions are compatible versions.
* Check for classpath conflicts with existing dependencies.

Missing input datasets for sink connectors::

* Verify that the `openlineage.integration.dataset.kafka.bootstrap.servers` property is configured.
* Verify that the connector has access to the Kafka bootstrap servers.
* Verify that the Kafka topics specified in the `topics` configuration exist and that the connector has access to them.
